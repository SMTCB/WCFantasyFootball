Message
Components

Usage guidelines for inline banners, toast notifications, system banners and mobile action sheets — covering all four semantic states: success, warning, error and informational.

01 — Semantic Types

Every message belongs to one of four semantic types. The type drives the colour, icon and tone of the message — never use colour alone to communicate meaning.

SUCCESS

Positive outcome

Confirms that a user action completed successfully and the system is in a good state. Should be brief and self-dismissing where possible.

  • Transfer confirmed / squad saved
  • Bid accepted on the market
  • Captain pick locked in
  • Payment or subscription activated
⚠ Avoid: using for neutral system events
WARNING

Attention needed

Something is at risk but not yet broken. The user can still act to prevent a negative outcome. Always include a clear CTA when possible.

  • Transfer deadline approaching
  • Captain pick likely to not start
  • Yellow card carry-over risk
  • Budget close to limit
⚠ Avoid: using for purely informational messages
ERROR

Action failed

A user-initiated action failed or the system encountered a critical problem. Always explain what went wrong and offer a recovery path.

  • Transfer failed (budget / locked)
  • Bid outbid on the market
  • Session expired or auth error
  • Network or sync failure
⚠ Avoid: using for warnings or deprecations
INFO

Neutral update

Communicates a neutral system event or useful context the user didn't trigger. No urgency. Should not interrupt the user's workflow.

  • Gameweek is now live / locked
  • Price changes scheduled
  • New league rules in effect
  • Feature announcements
⚠ Avoid: using for events that need action

03 — Format Decision Guide

Ask these questions in order to pick the right format:

1

Does the user need to take action?

Yes → use an Inline Banner (with actions) or Mobile Action Sheet. These are the only formats that persist until the user makes a choice. Never auto-dismiss a message that has an action button.

2

Is it a direct response to something the user just did?

Yes → use an Inline Banner (dismissible) placed immediately after the action element. This anchors feedback to context. If the action was in a modal or sheet, show the banner inside it, not on the page behind.

3

Is it a background event the user didn't trigger?

Yes → use a Toast. These are ephemeral and non-blocking. Keep the message to one line. If it's critical and requires action, escalate to an Inline Banner instead.

4

Does it affect the entire session / page?

Yes on desktop → use a System Banner at the top of the page. Yes on mobile → use a Mobile Action Sheet anchored to the bottom. These are reserved for states that change how the whole UI behaves (e.g., window closed, sync lost).

04 — Do's & Don'ts

✓ Do

  • Write message titles in sentence case (not ALL CAPS)
  • Use the JetBrains Mono type label as the only all-caps element
  • Keep body copy under 2 lines — link to detail if needed
  • Always pair error messages with a recovery action
  • Auto-dismiss success toasts after 4–5s
  • Stack toasts; limit to 3 visible at once
  • Use the semantic colour for border + icon only — not as fill
  • Test contrast: body copy must pass AA on ink-2 background

✕ Don't

  • Don't use warning for success or info for errors
  • Don't auto-dismiss a message that has an action button
  • Don't stack more than one system banner at a time
  • Don't use toasts for anything that requires a decision
  • Don't repeat the type label in the message title ("Error: Transfer failed" → "Transfer failed")
  • Don't use colour as the only differentiator — always include icon + label
  • Don't show inline banners outside the content column they relate to
  • Don't animate toasts with anything other than a simple fade/slide-in

05 — Copy Tone

Type Title tone Body tone Example title
Success Confident, affirming. What happened. Brief confirmation of outcome + relevant data point. Transfer confirmed
Warning Direct, urgent but calm. What is at risk. State the risk + time constraint if applicable. Don't catastrophise. Deadline in 18 minutes
Error Factual. What failed. No blame. Why it happened (briefly) + what the user can do next. Avoid technical jargon. Transfer failed
Info Neutral, matter-of-fact. What changed. Context only. No urgency. No call to action unless genuinely useful. Gameweek 29 now live

06 — Design Tokens

All message components are built exclusively from existing FORZAKIT tokens. No new colour values were introduced.

/* Semantic signal colours */
--positive: #22C55E; /* success state */
--warn : #F59E0B; /* warning state */
--danger : #EF4444; /* error state */
--cyan : #00B4D8; /* info / live */

/* Surface colours */
--ink : #080A0E; /* page background */
--ink-2 : #0F1218; /* card / panel bg */
--ink-3 : #161B25; /* toast background */
--rule : #1E2530; /* dividers / borders */

/* Foreground */
--paper : #F2EEE5; /* primary text */
--mute : #8B95A1; /* secondary text */

/* Usage pattern for tinted backgrounds */
/* bg : color-mix(in srgb, var(--signal) 9%, var(--ink-2)) */
/* bdr : color-mix(in srgb, var(--signal) 22%, transparent) */
/* icon: color-mix(in srgb, var(--signal) 14%, transparent) */
