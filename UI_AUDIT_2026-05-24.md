# 🔴 CRITICAL

## CROSS-CUTTING

### U1 — `SettingsScreen` destructures wrong auth method → users can never log out
- **File:** `src/screens/SettingsScreen.jsx:10` and `:49-57`
- **Issue:** `const { user, logout } = useAuth()` — but `AuthContext` exports `signOut`, not `logout`. Clicking "Logout" → `TypeError: logout is not a function` → "Error logging out" toast → user stays signed in. This is the **only** sign-out path in the app.
- **Fix:** Replace `logout` with `signOut` in the destructure and the call site.
- **Test:** Set `VITE_AUTH_ENABLED=true`, sign in, go to `/settings`, click Logout → should redirect to `/auth` with no error.

### U2 — Onboarding wizard renders over the login screen
- **File:** `src/App.jsx:38-57`
- **Issue:** `<OnboardingWizard>` is mounted in `AppRoutes` based purely on `showWizard` (localStorage flag). It sits **outside** `<ProtectedRoute>`. On a first cold visit with auth enabled, the wizard appears on top of `/auth`. After the user clicks the wizard's CTA (which navigates to `/market`), `ProtectedRoute` bounces them back to `/auth` — but now the wizard's `done` flag is set, so it never replays for the eventually-logged-in user.
- **Fix:**
  ```jsx
  const { user, authEnabled } = useAuth();
  {showWizard && (!authEnabled || user) && (
    <OnboardingWizard onComplete={completeWizard} onSkip={skipWizard} />
  )}
  ```
- **Test:** Clear localStorage with auth enabled → visit `/` → see only `/auth`, no wizard. Sign in → wizard fires.

### U3 — `/join?code=XXX` invite links 404 silently
- **File:** `src/components/LeagueInviteCard.jsx:48-52` generates `${BASE_URL}/join?code=${league.join_code}` (used in WhatsApp share + Copy Link). `src/App.jsx:84` catches `*` → redirects to `/`.
- **Issue:** Every shared invite from WhatsApp/SMS lands on home with no context. The join code is dropped silently. The entire invite mechanic doesn't work end-to-end.
- **Fix:** Add a `/join` route that reads `?code=` from query, pre-fills the join form on `LeagueScreen` (or directly calls `join_league_by_code` if authenticated). If unauthenticated, route to `/auth?returnTo=/join?code=...`.
- **Test:** Open `<prod>/join?code=ABCDEF` while signed in → should land on the league or pre-fill code field.

### U4 — `BrowserRouter` breaks Capacitor mobile cold launch + no Android back-button handler
- **Files:** `src/App.jsx:1, 98`; `src/lib/capacitor.js` (no `backButton` listener)
- **Issue:** Capacitor serves the app from `capacitor://localhost` (iOS) / `https://localhost` (Android). Only `/` maps to `index.html`. With `BrowserRouter`:
  - Cold-launching iOS to a non-`/` route → blank screen / 404 from webview.
  - Android hardware back at depth → app exits immediately.
- CLAUDE.md explicitly says "hash-based for Capacitor compatibility" — the code uses BrowserRouter regardless.
- **Fix:**
  ```jsx
  // App.jsx
  import { BrowserRouter, HashRouter } from 'react-router-dom';
  import { isNative } from './lib/capacitor';
  const Router = isNative ? HashRouter : BrowserRouter;
  ```
  And in `capacitor.js initNative()`:
  ```js
  App.addListener('backButton', () => {
    if (window.history.length > 1) window.history.back();
    else App.exitApp();
  });
  ```
- **Test:** `npx cap run ios` → backgrounding and reopening at depth must restore route. Android → hardware back from `/squad` → `/`.

### U5 — `useDeadlineCountdown` hardcodes `MATCHDAY_ID='md1'`; transfer lock UX is dead
- **Files:** `src/hooks/useDeadlineCountdown.js:29`; `src/screens/MarketScreen.jsx:157` (also `'md1'`); `src/components/TransferWindowBanner.jsx` (only mounted in `LeagueScreen`, never in Squad or Market).
- **Issue:** Canonical matchday IDs are `'426-rN'` (per CLAUDE.md). Neither hook nor MarketScreen query ever finds a row. `deadline.isLocked` is permanently false. Users can sell/buy/swap freely after deadline. The big red "Transfer Window Closed" banner cannot appear (literally unreachable code in `MarketScreen.jsx:352-370`).
- **Fix:**
  - Update `useDeadlineCountdown` to take `matchdayId` as parameter, default to the next upcoming `matchday_deadlines` row for the league's `tournament_id`.
  - Mount `<TransferWindowBanner {...useTransferWindow(activeLeague)} />` at the top of `SquadScreen` and `MarketScreen`.
  - Gate all action buttons (Sell, Buy, Sub In/Out, captain assignment, joker, formation reorder) on `deadline.isLocked`.
- **Test:** Set a past deadline → buttons disable across Squad + Market; banner shows red "Closed".

### U6 — `LiveScreen` polls every 5 min; no Realtime subscription anywhere
- **File:** `src/screens/LiveScreen.jsx:7, 500-504`
- **Issue:** `REFRESH_MS = 5 * 60 * 1000`. Goals scored at minute 23 don't appear until minute 28. The `calculate-scores` Edge Function broadcasts on a `scores` channel — nothing listens. Live UX promise ("every player every league") is broken.
- **Fix:** Subscribe to `match_events` INSERT and `player_match_stats` UPDATE filtered by active fixture IDs. Keep polling as a 30-60s safety net. Optional: listen on the existing `scores` broadcast channel.
- **Test:** Open Live screen, manually INSERT a `match_events` row in SQL → UI must update within ~1s.

### U7 — Joker chip is selectable in UI but does nothing in scoring
- **Files:** `src/screens/SquadScreen.jsx:165-173, 322` (Joker pick UI); `src/screens/RecapScreen.jsx:98` (`joker: null` hardcoded); `src/components/RecapCard.jsx:110-126` (Joker badge); `supabase/functions/calculate-scores/index.js` (zero references to `daily_jokers` or `joker_player_id`).
- **Issue:** Users burn a chip activation that has no scoring effect. Onboarding copy ("Daily Joker lives here — activate carefully") promises mechanics that don't exist.
- **Fix:** Either ship Joker scoring (read `daily_jokers` for the matchday, apply N× multiplier in `rollupSquads`, wire into Recap) **or** remove all Joker UI until the backend is ready. See LOGIC_AUDIT L1.5 for backend wiring.
- **Test:** Activate Joker on a player who scores 5 → squad total should reflect Joker bonus per chosen rule.

### U8 — Trade proposals are placebo toasts
- **File:** `src/screens/LeagueScreen.jsx:227-246` `validateAndSendProposal`
- **Issue:** Validates positions then calls only `showToast('Proposal sent!', 'success')`. No `supabase.from('trade_proposals').insert(...)`. Code has a `// TODO: deeper position-cap check` comment. Recipients never see the proposal.
- **Fix:** Either wire to `trade_listings` / `trade_proposals` table (whichever schema is canonical) or hide the "Broadcast Proposal" button with "Coming soon" until backend lands.
- **Test:** Click trade icon on a manager → broadcast → check DB: zero rows currently.

### U9 — Deep link `/league/<invalid-id>` hangs on "SYNCING..." forever
- **File:** `src/screens/LeagueScreen.jsx:313-368` `loadLeagueById`
- **Issue:** When `leagues.select(...).eq('id', id).single()` returns null (not found or RLS-denied), code doesn't redirect or surface error. `activeLeague` stays `null` but `view` is forced to `'detail'` → renders "SYNCING..." shell forever. Every tab is dead because every hook bails on `!activeLeague?.league_id`.
- **Fix:**
  ```js
  if (!lData) {
    showToast('League not found or you are not a member', 'error');
    navigate('/league', { replace: true });
    return;
  }
  ```
- **Test:** Visit `/league/00000000-0000-0000-0000-000000000000` → expect redirect with toast; today: stuck shell.

## SQUAD / MARKET / DRAFT

### U10 — `DraftRecoveryScreen` writes `matchday_id: 'current'` → squad won't match scoring pipeline
- **File:** `src/screens/DraftRecoveryScreen.jsx:214`
- **Issue:** Real matchday IDs are `'426-rN'`. Writing `'current'` means: (a) `calculate-scores` won't find this squad when keying by `matchday_id`; (b) SquadScreen fixture query `LIKE '${squad.matchday_id}%'` returns empty → 0 points displayed; (c) the `(league_id, user_id, matchday_id)` conflict key locks this `'current'` row forever, so future weeks never roll forward.
- **Fix:** Resolve active matchday from `matchday_deadlines` for the league's `tournament_id`; write a canonical ID.
- **Test:** Complete a draft recovery → check that the squad joins to real fixtures and scoring updates after FT.

### U11 — `SquadScreen.fetchSquad` returns most-recently-created squad regardless of matchday
- **File:** `src/screens/SquadScreen.jsx:176-183`
- **Issue:** No `matchday_id` filter. After multiple rollover rows accumulate, "most recent by created_at" is non-deterministic if migrations rewrite timestamps. Combined with U10's `'current'` rows, screen may show the wrong squad.
- **Fix:** Filter explicitly by the active matchday derived from `matchday_deadlines`.
- **Test:** Insert two squad rows with different `matchday_id`s and same `created_at` — currently arbitrary which renders.

## LIVE / RECAP

### U12 — `RecapScreen` falls back to `matchday_id='md2'`
- **File:** `src/screens/RecapScreen.jsx:38`
- **Issue:** `const matchdayId = squadRow.matchday_id || 'md2'` and `.like('id', '${matchdayId}%')`. Won't match `'426-rN'` IDs at all. New users with no `matchday_id` see "0 pts" instead of an empty state.
- **Fix:** Read the latest `matchday_deadlines` row where `deadline_at < now()`; show "No completed matchday yet" empty state if none.
- **Test:** User with no matchday → currently shows "0 pts" instead of empty UI.

### U13 — `RecapScreen` double-counts captain points
- **File:** `src/screens/RecapScreen.jsx:79, 92-97, 275-278`
- **Issue:** `totalPoints` comes from `fantasy_points.total` (already has captain × baked in by `calculate-scores`). Then the Captain row renders `×2 = ${captain.points * 2}`, and the Best Player calc uses raw `pointsMap[pid]` (base points). Total ≠ sum of per-player rows. Captain shows "10 pts ×2 = 20" but the total already counted the doubled value.
- **Fix:** Pick one convention — show base everywhere with explicit ×2 multiplier, OR show effective points with clear "(post-multiplier)" label. Update Best Player to use effective points.
- **Test:** Captain 10 base + non-cap 9 base → recap math should add up; currently mismatched.

## AUTH / ONBOARDING

### U14 — Password recovery mode silently clears on any subsequent auth event
- **File:** `src/context/AuthContext.jsx:54-58`
- **Issue:** `setRecoveryMode(event === 'PASSWORD_RECOVERY')` runs on **every** auth event. After PASSWORD_RECOVERY fires, the next TOKEN_REFRESHED flips it back to false. The `?type=recovery` query param keeps the AuthScreen showing the right tab via `initialTab`, but the flag itself is lost.
- **Fix:**
  ```js
  if (event === 'PASSWORD_RECOVERY') setRecoveryMode(true);
  else if (event === 'SIGNED_OUT')   setRecoveryMode(false);
  ```

### U16 — "Replay Tour" button uses wrong localStorage key

### U17 — `ProtectedRoute` redirect drops query params and hash
